Q. Difference between rdbms and dbms
A. Both are systems to store and manage data, but the difference is in structure, relationships, and capabilities.
	DBMS : Focus: data storage & retrieval
	A DBMS stores data in a simple format (files or tables) without enforcing relationships between them.
	No strict relationship between tables
	Supports single-user or small systems
	Less complex
	No or limited constraints
	example : Microsoft Access, dBase, FoxPro
	
	RDBMS : Based on relational model (tables + relations)
	An RDBMS stores data in tables (rows & columns) and enforces relationships between tables using keys.
	Data stored in multiple related tables
	Uses Primary Key & Foreign Key
	Supports constraints (ACID properties)
	Multi-user, scalable, secure
	Uses SQL (Structured Query Language)
	example : MySQL, Oracle Database, Microsoft SQL Server, PostgreSQL
	
	Feature				DBMS						RDBMS
	Data Structure		File-based / simple tables	Relational tables
	Relationships		❌ Not supported			✅ Supported (via keys)
	Normalization		❌ Limited					✅ Supported
	Constraints			❌ Minimal					✅ Strong (PK, FK, UNIQUE)
	Multi-user			Limited						Fully supported
	Security			Low							High
	Transactions		❌ Not fully supported		✅ ACID compliant
	Scalability			Small systems				Large enterprise systems
	
Q. name different type of indexes in mysql
A. A database index is a data structure that improves the speed of operations in a table. 
    Practically, indexes are also a type of tables, which keep primary key or index field and a pointer to each record into the actual table.
    The INSERT and UPDATE statements take more time on tables having indexes, whereas the SELECT statements become fast on those tables. 
    The reason is that while doing insert or update, a database needs to insert or update the index values as well.
    1. Unique - The Unique index in MySQL specifies that all values in the columns of a table must be distinct when applied. 
        single column : no duplicate value 
        multi column : grouping of column values in every row should be unique
        unique coumn can have single null value but it depends on language. ANSI SQL, Oracle, PostgreSQL allows columns with UNIQUE constraints to have multiple NULLs.
        “NULL” is not a value in itself and since it is not a value, it is not comparable at all, so it can be multiple.
        but NULL=NULL is true, so many dbms implements unique column with single null value like mysql,  Microsoft sql server
        CREATE UNIQUE INDEX Indexname ON Tablename(indexcolumn1, indexcolumn2,…);
        OR,
        CREATE TABLE Tablename(Column1 CHAR (30) NOT NULL, UNIQUE INDEX (column2));
        OR,
        ALTER TABLE Tablename ADD UNIQUE INDEX (column1, column2);
    2. Primary Key - uniquely identifies a row in tables. unique+not null = primary  key
        CREATE TABLE Tablename (Columnname Data_type PRIMARY KEY );
        OR,
        CREATE TABLE Tablename (Column1 CHAR(30) NOT NULL, Column2 CHAR(30) NOT NULL, PRIMARY KEY (Column1, Column2));
        OR,
        ALTER TABLE Tablename ADD PRIMARY KEY(Column1, Column2);
    3. Simple | Regular | Normal
        this type of Index named simple, regular, or normal, the specified column values do not require to be unique and can be NULL. 
        They essentially supplement the database searching process to make record retrieval faster
        CREATE INDEX Indexname ON Tablename  (Indexcolumn1, Indexcolumn2, …);
        OR,
        CREATE TABLE Tablename(Column1 CHAR(30) NOT NULL, INDEX(Column2));
        OR,
        ALTER TABLE Tablename ADD INDEX(Column1, Column2);
    4. Full-Text
        This index type is implemented for full-text searches. suppose you need to find a blob that includes a certain word or combination of words, 
        or it can even be a substring within the higher block text. 
        these indexes are maintained only for MyISAM and InnoDB tables in the database and can consist of only VARCHAR, CHAR, and TEXT table columns.
        CREATE FULLTEXT INDEX Indexname ON Tablename(indexcolumn1, indexcolumn2, …);
        OR,
        CREATE TABLE Tablename (column1 TEXT, FULLTEXT(column1));
        OR,
        ALTER TABLE Tablename ADD FULLTEXT(column1, column2);
    5. Spatial
        MySQL allows the creation of Spatial indexes on geometry-valued columns with NOT NULL constraint.
        The spatial index creates an R-tree index. The engine builds up a B-tree index for storage engines that maintain nonspatial indexing of the spatial table columns
        CREATE INDEX Indexname ON Tablename(Indexcolumn1);
        OR,
        CREATE TABLE Tablename (column1 GEOMETRY NOT NULL SRID 4326,SPATIAL INDEX(Column2));
        OR,
        ALTER TABLE Tablename ADD SPATIAL INDEX (column1, column2);
    6. Descending
        Version 8+ of MySQL offers the Descending index, a consistent index stored in the inverse order
        It supports users when they run queries to find out the most newly added data
        CREATE INDEX Indexname ON Tablename (Indexcolumn1 DESC);
        OR,
        CREATE TABLE Tablename (Column1 INT, Column2 INT, INDEX asc_index1(Column1 ASC, Column2 ASC), INDEX desc_index1 (Column1 DESC,      Column2 DESC));
        OR,
        ALTER TABLE Tablename ADD INDEX(column1 DESC, column2 ASC);

Q. explain acid property
A. In order to maintain consistency in a database, before and after the transaction, certain properties are followed. These are called ACID properties
    1. Atomicity: either the entire transaction takes place at once or doesn’t happen at all.  i.e. transactions do not occur partially.
    Atomicity is also known as the ‘All or nothing rule’. 
    It involves the following two operations. 
    —Abort: If a transaction aborts, changes made to the database are not visible. 
    —Commit: If a transaction commits, changes made are visible. 
    2. Consistency : databse must be consistent before and after the transection
    3. Isolation : multiple transection occurs independentally without interference 
    4. Durability : changes of a successfull transection occurs even if the system failure occurs

		);

		);

		SELECT name

		);

		FROM employees e

Q. Explain % and _ and like query
A. These are used for pattern matching in SQL queries—mainly with the WHERE clause.
	
	LIKE Operator : LIKE is used to search for a specific pattern in a column.
		SELECT * FROM users WHERE name LIKE 'pattern';
	
	% (Percent Wildcard) : Represents zero, one, or multiple character
		SELECT * FROM users WHERE name LIKE 'A%';	-- Names starting with 'A'
	
	_ (Underscore Wildcard) : Represents exactly one character
		SELECT * FROM users WHERE name LIKE 'R___';		-- Names with exactly 4 letters starting with 'R'
	
	Escape Special Characters : If your data contains % or _, use ESCAPE
		SELECT * FROM products WHERE name LIKE '100\%' ESCAPE '\';
		this query will ignore \ and search for 100% in name column

Q. Mysql pdo conection and query
A. PDO : PHP Data Objects 
	We use PDO to connect to MySQL using a DSN string. Queries are executed using query() for simple cases or prepare() and execute() for prepared statements, which prevent SQL injection. PDO also supports transactions and multiple database drivers.
	
	<?php
	$host = "localhost";
	$db   = "test_db";
	$user = "root";
	$pass = "password";
	$charset = "utf8mb4";

	$dsn = "mysql:host=$host;dbname=$db;charset=$charset";

	$options = [
		PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION, // throw exceptions
		PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,       // fetch as assoc array
		PDO::ATTR_EMULATE_PREPARES   => false,                  // use real prepared stmts
	];

	try {
		$pdo = new PDO($dsn, $user, $pass, $options);
		echo "Connected successfully";
	} catch (PDOException $e) {
		die("Connection failed: " . $e->getMessage());
	}
	
	2) SELECT Query (Fetch Data)
	$stmt = $pdo->query("SELECT id, name FROM users");

	$users = $stmt->fetchAll();

	foreach ($users as $user) {
		echo $user['id'] . " - " . $user['name'] . "<br>";
	}
	3) Prepared Statement (Recommended) : Prevents SQL injection
	$stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?");
	$stmt->execute([1]);

	$user = $stmt->fetch();
	print_r($user);
	
	4) INSERT Query
	$stmt = $pdo->prepare("INSERT INTO users (name, email) VALUES (?, ?)");
	$stmt->execute(["Amit", "amit@example.com"]);

	echo "Inserted ID: " . $pdo->lastInsertId();
	
	5) UPDATE Query
	$stmt = $pdo->prepare("UPDATE users SET name = ? WHERE id = ?");
	$stmt->execute(["Rahul", 1]);
	
	6) DELETE Query
	$stmt = $pdo->prepare("DELETE FROM users WHERE id = ?");
	$stmt->execute([1]);
	
	7) Transactions (Very Important)
	try {
		$pdo->beginTransaction();

		$pdo->exec("INSERT INTO users (name) VALUES ('User1')");
		$pdo->exec("INSERT INTO users (name) VALUES ('User2')");

		$pdo->commit();
	} catch (Exception $e) {
		$pdo->rollBack();
		echo "Failed: " . $e->getMessage();
	}
	8) Named Parameters (Cleaner)
	$stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email");
	$stmt->execute(['email' => 'amit@example.com']);

	$user = $stmt->fetch();
	
Q. mysql what is cursor 
A. cursor is a database object used to process query results row-by-row instead of all at once.
	with curdor : 
	You process each row individually
	Useful in stored procedures
	
	Steps of using a cursor
		DECLARE cursor
		OPEN cursor
		FETCH rows
		PROCESS logic
		CLOSE cursor

	example : 
	DECLARE done INT DEFAULT FALSE;
	DECLARE user_name VARCHAR(100);

	DECLARE cur CURSOR FOR 
		SELECT name FROM users;

	DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

	OPEN cur;

	read_loop: LOOP
		FETCH cur INTO user_name;

		IF done THEN
			LEAVE read_loop;
		END IF;

		-- Process each row
		SELECT user_name;

	END LOOP;

	CLOSE cur;
Q. what function and procedure in mysql. explain with example
A. Function : A function is a reusable program (like a small block of logic) that takes input, processes it, and returns a single value.

	There are two types of functions in MySQL:
	Built-in functions (already available)
	User-defined functions (UDFs) (created by you)
	
	1. Built-in Functions (Most Common) : These are already provided by MySQL.
	example : SELECT UPPER('gautam');
		SELECT ROUND(12.567, 2);
		SELECT NOW();
		
	2. User-Defined Function (Custom Function) : You can create your own function using CREATE FUNCTION.
	DELIMITER //

	CREATE FUNCTION add_numbers(a INT, b INT)
	RETURNS INT
	DETERMINISTIC
	BEGIN
	   RETURN a + b;
	END //

	DELIMITER ;
	
	call function : SELECT add_numbers(5, 10);
	
	Points : 
	Function always returns a single value
	Used inside queries (SELECT, WHERE, etc.)
	Cannot return multiple rows (use stored procedure for that)
	Improves code reuse and readability

	procedure : a stored procedure is a saved block of SQL code that you can execute whenever needed. It’s used to perform operations like insert, update, validation, loops, etc.
	
	A stored procedure:
	Can take input parameters
	Can return output values
	Can execute multiple SQL statements
	may or may not have to return a value
	
	Example 1: Simple Procedure
	DELIMITER //

	CREATE PROCEDURE get_users()
	BEGIN
	   SELECT * FROM users;
	END //

	DELIMITER ;
	
	CALL get_users();
	
	Example 2: Procedure with Input Parameter
	DELIMITER //

	CREATE PROCEDURE get_user_by_id(IN user_id INT)
	BEGIN
	   SELECT * FROM users WHERE id = user_id;
	END //

Q.	Stored Procedure with Cursor
A. DELIMITER //

	CREATE PROCEDURE update_bonus()
	BEGIN
	   DECLARE done INT DEFAULT 0;
	   DECLARE emp_id INT;
	   DECLARE emp_salary INT;

	   -- Cursor declaration
	   DECLARE emp_cursor CURSOR FOR
		  SELECT id, salary FROM employees;

	   -- Handler for end of rows
	   DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = 1;

	   -- Open cursor
	   OPEN emp_cursor;

	   read_loop: LOOP
		  -- Fetch row
		  FETCH emp_cursor INTO emp_id, emp_salary;

		  IF done = 1 THEN
			 LEAVE read_loop;
		  END IF;

		  -- Apply logic
		  IF emp_salary > 50000 THEN
			 UPDATE employees 
			 SET bonus = emp_salary * 0.10
			 WHERE id = emp_id;
		  ELSE
			 UPDATE employees 
			 SET bonus = emp_salary * 0.05
			 WHERE id = emp_id;
		  END IF;

	   END LOOP;

	   -- Close cursor
	   CLOSE emp_cursor;

	END //

	DELIMITER ;

Q. what is the difference between group by and  having clause?
    Having Clause is basically like the aggregate function with the GROUP BY clause. 
    The HAVING clause is used instead of WHERE with aggregate functions. While the GROUP BY Clause groups rows that have the same values into summary rows. The having clause is used with the where clause in order to find rows with certain conditions. The having clause is always used after the group By clause.

    SELECT COUNT (SALARIES) AS COUNT_SALARIES, EMPLOYEES
    FROM EMPLOYEES
    GROUP BY SALARIES
    HAVING COUNT(SALARIES) > 1;
